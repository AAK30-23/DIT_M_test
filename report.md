## Часть 1. Бизнес-анализ

### Пространственная коллизия

Под пространственной коллизией я понимаю ситуацию, когда геометрии двух работ пересекаются полностью или частично. В рамках данного задания пространственная коллизия определяется на основе пересечения геометрических объектов, представленных в формате WKT. Если геометрии не имеют общей области пересечения, то пространственная коллизия отсутствует независимо от сроков выполнения работ.

### Временная коллизия

Временная коллизия возникает в случае пересечения периодов выполнения работ. Для двух работ с интервалами [start_1; end_1] и [start_2; end_2] временная коллизия считается существующей, если имеется хотя бы один общий день выполнения работ. Если временные интервалы не пересекаются, временная коллизия отсутствует.

### Комбинированная коллизия

Комбинированная коллизия возникает одновременно при выполнении двух условий:

* работы пересекаются в пространстве;
* работы пересекаются по времени.

Именно комбинированная коллизия рассматривается как потенциальный конфликт, поскольку может приводить к конфликтам между подрядчиками, повторному выполнению работ на одном участке и дополнительным затратам.

### Какие коллизии считать критичными

Критичными предлагаю считать комбинированные коллизии, удовлетворяющие хотя бы одному из условий:

* хотя бы одна работа имеет статус in_progress;
* хотя бы одна работа имеет высокий приоритет (2 или ниже);
* продолжительность временного пересечения превышает 7 календарных дней.

Такие ситуации требуют дополнительной проверки, поскольку вероятность реального конфликта между работами наиболее высока.

### Учет статуса работ

Статус работ влияет на оценку критичности коллизии.

* planned - участвует в анализе;
* in_progress - участвует в анализе и повышает критичность;
* done - участвует только для анализа исторических пересечений;
* canceled - не рассматривается как критичная коллизия (работы со статусом canceled сохраняются в данных, но не учитываются при поиске актуальных конфликтов).

### Учет приоритета

Приоритет используется как дополнительный фактор при оценке серьезности коллизии. Чем выше приоритет пересекающихся работ, тем выше потенциальные последствия конфликта.

* 1 - очень высокий;

* 2 - высокий;

* 3 - средний;

* 4 - низкий.

Для оценки силы коллизии используется показатель:

`Conflict Score = SpatialOverlap × TimeOverlap × PriorityWeight`

где:

* SpatialOverlap - доля пространственного пересечения;

* TimeOverlap - доля временного пересечения;

* PriorityWeight - среднее значение весовых коэффициентов двух работ (приоритет 1 соответствует весу 4, приоритет 2 - весу 3, приоритет 3 - весу 2, приоритет 4 - весу 1).

При отсутствии значения приоритета используется минимальный приоритет, равный 4.

### Обработка некорректных и неполных данных

При просмотре датасета были обнаружены несколько проблем качества данных:

* отсутствующие значения подрядчика (contractor);
* отсутствующие значения приоритета (priority);
* отсутствующие даты окончания работ (end_dt);
* дублирующиеся идентификаторы (A-003);
* некорректные интервалы дат (B-002 дата начала больше даты окончания);
* отсутствующая геометрия (A-006);
* разные варианты записи одного и того же адреса.

Для обработки таких случаев приняты следующие правила:

| Проблема               | Способ обработки                                              |
| ---------------------- | ------------------------------------------------------------- |
| Отсутствует геометрия  | объект исключается из пространственного анализа               |
| Отсутствует contractor | заменяется значением Unknown                                  |
| Отсутствует priority   | используется значение 1                                       |
| Отсутствует end_dt     | объект исключается из временного анализа                      |
| start_dt > end_dt      | даты меняются местами                                         |
| Дубликаты              | дубликаты рассматриваются как независимые записи |
| Разные форматы адресов | адрес не используется в анализе                              |

### Принятые допущения

1. Геометрии из полей geom_a_wkt, geom_b_wkt и geom_c_wkt описывают объекты в одной системе координат и могут анализироваться совместно после объединения в единое геометрическое поле.

2. Наличие пространственной коллизии определяется фактом пересечения геометрий независимо от типа работ.

3. Работы со статусом canceled не рассматриваются как актуальные конфликты.

4. Дубликаты рассматриваются как независимые записи, поскольку могут соответствовать разным версиям одной работы с различными статусами или сроками выполнения.

5. При отсутствии приоритета используется минимальный приоритет.

6. Объекты без геометрии не участвуют в пространственном анализе.

7. Пространственный анализ проводится по геометрическим данным без использования адреса.

### Выявленные проблемы

В ходе анализа были обнаружены следующие особенности исходных данных.

**1. Дубликаты**

В наборе данных **A** присутствуют две записи с идентификатором **A-003**. В рамках пространственного анализа они рассматриваются как независимые записи.

**2. Некорректные временные интервалы**

У записи **B-002** дата начала работ 08.03.2026 больше даты окончания 05.03.2026. Перед выполнением анализа эту запись нужно исправить или исключить.

**3. Пропуски**

В датасете встречаются пропуски в нескольких полях contractor, priority, end_dt, geom_a_wkt.

**4. Формат адресов**

Один и тот же адрес записан несколькими способами (Улица Косыгина; ул. Косыгина; ул Косыгина, Москва; Косыгина ул; Kosygina st.; УЛ.КОСЫГИНА).

**5. Геометрия**

Каждый источник использует собственное поле хранения геометрии (набор A - geom_a_wkt; B - geom_b_wkt;  C - geom_c_wkt). Перед пространственным анализом данные нужно объединитьв одно общее поле геометрии.

### Выводы

Проверка показала, что исходные данные содержат дубликаты, пропуски и некорректные значения дат. Большинство ошибок могут быть устранены на этапе предварительной обработки данных и не повлияют на дальнейший анализ. Особенностью датасета является хранение геометрии в разных столбцах в зависимости от источника данных, поэтому одним из этапов обработки является объединение геометрии в общее поле, которое затем используется для проверки пространственных пересечений.


## Часть 3. Пространственный анализ

### 3.1 Подготовка пространственных данных

В исходном датасете геометрия объектов хранится в разных столбцах geom_a_wkt, geom_b_wkt, geom_c_wkt в зависимости от источника данных A, B или C. Поэтому данные были объеденены в поле geometry, содержащее геометрию объекта независимо от его источника.

Дальше были сформированы три пространственных слоя и преобразованы в объект GeoDataFrame, чтобы выполнять пространственные операции с библиотекой GeoPandas:

* **Слой A** - записи с dataset_code = A;
* **Слой B** - записи с dataset_code = B;
* **Слой C** - записи с dataset_code = C.

### 3.2 Проверка геометрии
* наличие геометрии;
* корректность геометрии is_valid;
* система координат.

Исходные геометрии представлены в системе координат WGS84 (EPSG:4326). После загрузки они были перепроецированы в метрическую систему координат EPSG:28406, используемую для корректного расчета площадей пересечения и коэффициента SpatialOverlap. Записи без геометрии были исключены из анализа.

### 3.3 Поиск пространственно-временных коллизий

Коллизии искались между следующими парами наборов данных:

* A * B;
* A * C;
* B * C.

Для каждой пары объектов последовательно выполнялись следующие проверки:

1. пересекаются ли геометрии объектов;
2. пересекаются ли периоды выполнения работ;
3. определяется количество дней временного перекрытия;
4. рассчитывается показатель силы коллизии.

### 3.4 Расчет показателей

Количество дней перекрытия вычисляется как пересечение временных интервалов двух работ. Для оценки значимости найденных коллизий используется показатель Conflict Score. Использование относительных величин позволит корректно сравнивать коллизии между объектами разных размеров и продолжительности.

### 3.5 Результат

По результатам анализа сформирована итоговая таблица коллизий collisions.csv, которая используется для ранжирования найденных коллизий и последующего анализа.

| work_type_1           | тип работ первой работы                       |

| object_2              | идентификатор второй работы                   |

## Часть 4. Аналитическая витрина

После определения пространственных коллизий была сформирована аналитическая витрина, содержащая агрегированные показатели для последующей визуализации в BI-системе. Витрина позволяет быстро оценить количество конфликтов, их распределение и наиболее проблемные участки.

### Логика формирования

На основе таблицы collisions.csv были рассчитаны следующие показатели:

* количество коллизий по типам пар (A*B, A*C, B*C);
* распределение конфликтов по типам;
* количество критичных коллизий;
* среднее и максимальное количество дней временного перекрытия;
* распределение по статусам работ;
* распределение по уровням приоритета.

Все расчеты выполняются в Python. Итоговая витрина сохраняется в файл datamart.csv, который может использоваться в BI-системах.

## Часть 5. Дополнительные предложения

### 5.1 Визуализация результатов

Полученная аналитическая витрина может использоваться в BI-системах (Power BI, Tableau, Apache Superset) для построения интерактивных дашбордов. Использованы следующие визуализации:

| Визуализация         | Назначение                                                                                             |
| -------------------- | ------------------------------------------------------------------------------------------------------ |
| Столбчатая диаграмма | Количество коллизий по парам A*B, A*C, B*C                                                             |
| Круговая диаграмма   | Распределение по статусам работ                                                                        |
| Столбчатая диаграмма | Распределение по приоритетам                                                                  |
| Карточки KPI         | Общее количество коллизий, число критичных коллизий, среднее и максимальное количество дней перекрытия |


### 5.2 Предлагаемая архитектура промышленного решения

Для промышленной эксплуатации предлагается следующая архитектура.

Источники данных -> ETL -> Python / Airflow -> PostGIS -> Пространственный анализ -> Аналитическая витрина -> Power BI / Superset

В такой архитектуре:

* исходные данные регулярно загружаются в PostGIS;
* пространственные функции выполняются средствами SQL;
* результаты сохраняются в аналитическую витрину;
* BI-система автоматически отображает актуальные показатели.

Использование PostGIS позволит ускорить поиск пространственных пересечений по сравнению с последовательной обработкой объектов в Python.

### 5.3 Автоматический мониторинг коллизий

Для постоянного контроля предлагается автоматизировать процесс проверки. После появления новых данных выполняются следующие действия:

1. загрузка новых объектов;
2. проверка качества данных;
3. поиск новых пространственно-временных коллизий;
4. обновление аналитической витрины;
5. уведомление ответственных сотрудников при появлении критичных конфликтов.

В качестве средства автоматизации может использоваться Apache Airflow. При обнаружении критичной коллизии возможно автоматическое создание уведомления по электронной почте либо через корпоративный мессенджер.
